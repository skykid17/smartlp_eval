Build a Retrieval-Augmented Generation (RAG) system that takes in a log, analyzes its type, and recommends an integration package from the Elastic integration repository if a match is found.

Tech Stack:
- Python for implementation
- Ollama to host the LLM and embedding model
- Chroma as the vector database

Precompute
- Embeddings of known log_types (package directory name) and their sample logs (inside the corresponding directories) from the Elastic integration repository stored locally in "C:\Users\geola\Documents\GitHub\elastic_integrations". Have the LLM find the log samples within the repository.
Functional Approach:

- Accept incoming log as input (string format).
- Let LLM identify log_type using the following prompt:
"""Determine the specific log type from a given log entry. 
Analyze the log format, including timestamp style, field structure, severity labels, or known keywords.
Use domain-specific clues (e.g., paths, methods, IPs for web logs; query/error keywords for databases).
Match the log entry to one of the predefined types based on these patterns.
Return the exact log type identifier (e.g., "nginx_access").
If the log does not clearly match any type or you're unsure, return "unknown"."""

- Generate embeddings of the log using Ollama's embedding model.
- Store precomputed 
- Compare incoming log embeddings against stored log sample embeddings to find the closest match.
- If a match is found
    - return steps to install the corresponding Elastic integration package. (also found in the repository, can provide link to elastic official documentation for the integration)
- If no match is found
    - return "no package found"

- Use modular functions instead of classes.
- Ensure each function has a single responsibility.
- Optimize for readability and maintainability.
